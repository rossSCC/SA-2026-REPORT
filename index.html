Mission Status

The Smart Forest Sentinel meets the 2026 brief by providing a solution for the Forests theme, specifically targeting Wildfire Risk Prediction. The system is split into two phases: an autonomous field unit and an analytical command suite.

Advanced Requirement Summary

  • **Adaptive System:** Implemented via the Dry Day Counter, which dynamically adjusts the fire risk sensitivity based on sustained heat and dryness.
  • **Risk Modelling:** A Python model is used to analyze logged sensor data and project future risk via **"What-If"** scenarios.
  • **Process Simulation:** The Micro:bit's algorithm simulates the accumulation of fuel dryness, which is the primary cause of major wildfires.

RESEARCH LOG: FIRE WEATHER INDEX

Environmental Drivers of Risk

Research showed that instantaneous temperature is less critical than the **cumulative drying** of forest fuels. To accurately model this, the embedded system needed a proxy for moisture/humidity, which led to using the Light Sensor reading as a solar radiation proxy.

Constraints and Technology Selection

The choice of the Micro:bit's internal sensors and the **24-LED Halo** was justified by the need for a simple, rugged, and low-power autonomous system, prioritizing the code complexity (for high marks) over overly complex hardware setup.

SYSTEM ARCHITECTURE: FIELD DEPLOYMENT

The Store and Dump Protocol

To ensure the Micro:bit can operate autonomously in the field without a constant computer connection, a **Store and Dump** protocol was designed. Data is continuously written to the Micro:bit's internal Flash Memory and retrieved only when the device is connected and Button B is pressed.

[MICROBIT] --(BATTERY POWERED)--> [FLASH LOG: data.txt]
[USER INPUT: Button B] --(TRANSFER)--> [SERIAL CONSOLE]
[PC] -> [PYTHON MODEL]

Embedded System Flow

The system logic begins with a Calibration Phase (Button A), where a baseline light level is set. The main loop then runs the adaptive risk calculation and updates the Halo output, providing the hiker with a clear visual warning.

Problem & Resolution: Sensor Variance

**Problem:** Readings were inconsistent depending on the starting location (e.g., inside a classroom vs. outside).
**Resolution:** A **Calibration Mode** was implemented. The system now uses the light level recorded at the moment Button A is pressed as a dynamic baseline, ensuring the risk calculation is relative to the device's specific deployment environment.

FINAL EVALUATION: MISSION COMPLETE

Artefact Assessment

The **Firewatch Console** system is highly successful. The separation of the logging and analysis tasks ensures both robustness and complexity. The **Adaptive System** exceeded the basic requirement by simulating real-world processes.

Future Iterations

Improvements would focus on connectivity and precision:

  1. Real-Time Data: Integrate a **LoRaWAN module** for long-range, low-power telemetry, replacing the manual data dump.
  2. Precision: Add a dedicated **Humidity Sensor** to provide the core data required for professional-grade FWI calculation.
  3. Data Integrity: Add an **RTC (Real-Time Clock)** to log accurate timestamps, crucial for temporal analysis.
